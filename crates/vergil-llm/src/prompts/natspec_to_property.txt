The author wrote these comments to document intended behavior; your job is to convert the documented intent into formal properties a downstream synthesizer can translate into Halmos. You are NOT writing Halmos here — you are stating each invariant in precise English.

## Tag treatment (READ BEFORE GENERATING)

Each NatSpec tag carries a different signal strength. Honor these distinctions:

- **`@invariant`** is the strongest signal — the author explicitly named an always-true property. Translate it as DIRECTLY as possible into an English invariant statement. Do not paraphrase aggressively; preserve the author's quantifiers, comparison operators, and named contract terms.
- **`@custom:security`** is also a strong signal — typically used by audited contracts to encode intentional security invariants. Treat it the same as `@invariant`: direct, minimal reinterpretation.
- **`@notice`** and **`@dev`** describe behavior but may be informal or aspirational. Identify the underlying invariant and generalize. Be wary: doc comments can drift from actual code, and an author may document buggy behavior as "expected".

When multiple tag types are present on one block, prefer extracting from `@invariant` / `@custom:security` first, and only generate additional candidates from `@notice` / `@dev` when they capture distinct properties.

## Task

Identify between 1 and {{ max_candidates }} candidate invariants documented in this NatSpec block. For each, produce:

- `name`: a `snake_case` identifier suitable for a Halmos `check_` function (do NOT include the `check_` prefix — the synthesizer adds it). Derive it from the target name and the property — e.g. `transfer_preserves_total_supply`, `only_owner_can_pause`.
- `intent_text`: a single English sentence stating the invariant *as a universal property the contract should always satisfy*. Be concrete and verifiable. For `@invariant` / `@custom:security`, stay close to the author's wording. For `@notice` / `@dev`, generalize.
- `rationale`: one sentence explaining which tag this came from and how the invariant was extracted (literal vs paraphrased vs generalized). This is for the audit trail.

If the block carries only `@invariant` tags, all extracted properties should match those tags directly — one property per invariant unless one invariant decomposes naturally into multiple. Do not invent properties the doc comment doesn't support.

If the block is too vague to generalize (e.g., a function-level `@notice` saying "returns the balance"), return an empty list.

## Rules

1. **Direct treatment for `@invariant` / `@custom:security`.** Match the author's wording closely. The downstream synthesizer is good at translating precise English into Halmos; over-paraphrasing here loses information.
2. **Don't paraphrase code as a property.** A doc comment saying "sets `owner` to `newOwner`" is documenting the implementation, not an invariant. The corresponding invariant would be "after `transferOwnership(newOwner)` succeeds, `owner == newOwner`". The downstream critique rejects spec-by-paraphrase.
3. **Reject buggy intent.** If the doc comment appears to describe buggy behavior (e.g., "overflow is allowed"), do not invent a property from it — the verifier needs to be able to catch the bug.
4. **Vary across candidates.** When extracting multiple invariants from one block, each must be meaningfully distinct.
5. **Match Halmos's verification surface.** Properties must be expressible in symbolic execution on a single sequence of contract calls (no `vm.*` cheatcodes, no time travel). Skip properties that require multi-block or multi-actor reasoning.
6. **Storage-level invariants when applicable.** If the block is attached to a storage variable (e.g., `uint256 public totalSupply` with `@invariant totalSupply == sum(balanceOf[*])`), extract the variable-level property. The synthesizer can quantify across mappings and external functions.

If no useful invariant can be extracted, return `[]`.
