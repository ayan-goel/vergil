You are NOT writing Halmos here — you are stating the invariant in plain English.

## Task

Read the test and identify between 1 and {{ max_candidates }} candidate invariants the contract should always satisfy. For each candidate, produce:

- `name`: a `snake_case` identifier suitable for a Halmos `check_` function (do NOT include the `check_` prefix — the synthesizer adds it).
- `intent_text`: a single English sentence that states the invariant *as a universal property the contract should always satisfy*, not as a description of what this specific test happens to do. The synthesizer treats this as the intent and is responsible for turning it into Halmos. Be concrete and verifiable; avoid hedging words like "should usually" or "in most cases".
- `rationale`: one sentence explaining how the invariant generalizes from the test's assertions. This is for the audit trail — what in the test convinced you the property holds in general.

If the test is too narrow or buggy to generalize a property from (e.g., a smoke test that only asserts `1 == 1`, or a regression test for a specific bug fix that doesn't expose a general invariant), return an empty list.

## Rules

1. **Do not restate the test's literal assertion.** A test asserting `assertEq(balanceOf(alice), 100)` is checking "transfer is correctly accounted for"; generalize that, not "alice has 100 tokens after transfer". The downstream critique pass will reject literal restatements as vacuous.
2. **Generalize over symbolic inputs.** Replace the test's concrete amounts / addresses with universal quantifiers ("for any amount", "for any sender") when stating the invariant.
3. **State invariants in terms of contract state, not just call results.** "transfer returns true on success" is weaker than "transfer preserves totalSupply". Prefer state-level invariants when the test's assertions exercise contract state.
4. **Reject buggy intent.** If the test appears to encode buggy behavior as expected (e.g., a test asserting that an overflow silently wraps), do not invent a property from it; the critique pass needs to be able to catch the bug, not enshrine it.
5. **Vary across candidates.** When returning multiple candidates from one test, each must capture a meaningfully different invariant. Do not generate near-duplicates of the same property phrased two ways.
6. **Match Halmos's verification surface.** Properties must be expressible in symbolic execution on a single sequence of contract calls (no `vm.*` cheatcodes, no time travel, no `msg.sender` shifting). If the test relies on multi-actor scenarios or `vm.prank`, only extract properties that hold for the single-caller case.

## Output schema

Return a single JSON array. No prose. No code fences. The schema is:

```json
[
  {
    "name": "snake_case_identifier",
    "intent_text": "a single English sentence stating the invariant",
    "rationale": "one-sentence explanation of how this generalizes from the test"
  }
]
```

If no useful invariant can be generalized from the test, return `[]`.
