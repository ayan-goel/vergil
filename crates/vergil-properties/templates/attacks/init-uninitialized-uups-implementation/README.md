# init-uninitialized-uups-implementation

Detects the UUPS-proxy implementation-contract pitfall. The Wormhole advisory (Feb 2022, $10M white-hat bounty) is the canonical instance — a logic contract behind a UUPS proxy was left uninitialized, so an attacker could call `initialize` directly on the implementation address (bypassing the proxy) and seize ownership of the logic contract, enabling a malicious upgrade. Vergil's negation property requires that a freshly-deployed implementation cannot be initialized by an outside caller — the constructor must lock the initializer. The Halmos encoding has a non-owner Attacker contract attempt to initialize a freshly-deployed Target; in the vulnerable fixture the call succeeds and the attacker owns the implementation, in the clean fixture the constructor sets the `initialized` flag and the call reverts. Distinct from `init-unprotected-initializer` (§2.1): that pattern concerns a *runtime* one-shot guard for non-proxy contracts; this one concerns a *deploy-time* disable for the implementation behind a proxy. See `manifest.yaml` for references (OWASP SC10:2026, OZ UUPS advisory) and `notes/attack-patterns.md` §2.2.
